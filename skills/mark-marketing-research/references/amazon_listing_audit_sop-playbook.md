# Amazon Listing Audit SOP Playbooks\n\n## Supabase Query Fallback Procedure\n\nThis procedure is MANDATORY whenever querying Supabase for any data (SOPs, audit results, phase data, or any stored content). The hard rule: **never exceed 2 query attempts for the same data request.**\n\n### Step 1 — Verify Table Existence\n1. Call `list_tables` to get all available tables in the database.\n2. Check if the target table exists. Match the exact table name (case-sensitive).\n3. If the table does NOT exist: stop immediately. Tell the user: \"The table [name] does not exist in the database. Here are the available tables: [list].\" Offer to check if the data might be in a different table.\n4. If the table exists: proceed to Step 2.\n\n### Step 2 — First Query Attempt (Targeted)\n1. Write a targeted SQL query using explicit column names (avoid `SELECT *` on large tables).\n2. Use `LIMIT 10` unless the user needs all rows.\n3. Execute the query via `execute_sql`.\n4. If results are returned: deliver the data to the user. Done.\n5. If empty results or error: proceed to Step 3.\n\n### Step 3 — Diagnostic Query\n1. Do NOT immediately retry the same query.\n2. Run a diagnostic query to understand why the first attempt failed:\n   - `SELECT count(*) FROM table_name;` — Is the table empty?\n   - `SELECT * FROM table_name LIMIT 5;` — What does the data actually look like?\n   - `SELECT column_name, data_type FROM information_schema.columns WHERE table_name = 'target_table';` — What columns exist?\n3. Analyze the diagnostic results:\n   - If count is 0: the table exists but is empty.\n   - If columns don't match expectations: the schema is different than assumed.\n   - If sample rows exist: the original WHERE clause was too restrictive.\n\n### Step 4 — Second Query Attempt (Adjusted)\n1. Based on diagnostic findings, adjust the query:\n   - Fix column names to match actual schema.\n   - Broaden WHERE clause if too restrictive.\n   - Try a different column for filtering.\n2. Execute the adjusted query.\n3. If results are returned: deliver the data. Done.\n4. If still empty or error: proceed to Step 5.\n\n### Step 5 — Clear Exit (MANDATORY after 2 failed attempts)\n1. **Stop querying.** Do not attempt a third query.\n2. State the specific technical finding clearly. Use one of these patterns:\n   - \"The table [name] exists but contains 0 rows — it appears to be empty.\"\n   - \"The table [name] exists with [N] rows, but none match [criteria]. The available data looks like: [brief sample].\"\n   - \"The query returned an error: [specific error]. This suggests [interpretation].\"\n3. Offer exactly these alternatives:\n   - \"I can check if this data might be stored in a different table — here are the other tables available: [list].\"\n   - \"You can check the Supabase dashboard directly at your project URL to browse the table contents.\"\n   - \"If you can paste the content here, I'll work with it directly.\"\n4. Let the user choose. Do not keep trying on your own.\n\n### What NOT to Do\n- ❌ Say \"let me try\" without explaining what you'll do differently\n- ❌ Attempt 3+ queries for the same data request\n- ❌ Assume table names without calling `list_tables` first\n- ❌ Blame vague \"access restrictions\" — state what you actually observed\n- ❌ Use `SELECT *` on tables that might be large\n- ❌ Keep the user waiting through a loop of failed attempts\n
